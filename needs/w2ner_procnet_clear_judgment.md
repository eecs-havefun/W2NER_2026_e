## 逐项明确判断

### 1. 关于原始 W2NER 输出

**判断：你的 W2NER 输出已经包含足够的实体定位信息，可以支撑转换到 ProcNet。**

理由：

- `doc_id`、`sent_id`、`sentence`、`procnet_entities` 都齐全
- `procnet_entities` 中已经给出：
  - `b`
  - `e`
  - `type`
  - `text`
- 这些字段已经足够构造 ProcNet 侧的：
  - `ann_mspan2dranges`
  - `ann_mspan2guess_field`

**结论：这一项没有结构性问题。**

---

### 2. 关于转换后的 ProcNet 样本

**判断：你的 ProcNet 样本格式不是“简单照搬官方原始 key”，而是做了“增强 key”设计；这个设计本身可行，但要确认下游代码是否始终按这个约定读取。**

你现在使用的 key 形式是：

```text
实体文本#sentIdx_start_end#类型
```

例如：

```text
春秋航空#0_1_5#orderApp
```

这意味着：

- 你不是用“纯文本 span”做 key
- 而是用“文本 + 位置 + 类型”的复合 key

这样做的优点：

- 可以避免同一文本在不同位置重复时冲突
- 可以避免同形异类时互相覆盖

这样做的代价：

- 下游任何地方如果默认把 key 当成“原始 span 文本”，就会出语义错位
- `recguid_eventname_eventdict_list` 里填的是纯文本值，而 `ann_mspan2dranges` 的 key 却是复合 key，这要求读取阶段必须有一致的映射逻辑

**结论：这个设计不是错，但它不是官方最朴素写法。它要求你的修改版 ProcNet 全链路都接受这种复合 key。**

---

### 3. 关于 `b/e` 的索引口径

**判断：这一项已经对齐，没有问题。**

你补充的信息表明：

- `sentence` 是字级列表
- `b/e` 是字符级索引
- 同时也等于 token 索引
- 索引对象是 `"".join(sentence)`

这与 ProcNet 使用 `doc.sentences[sent_idx][start:end]` 的切片逻辑是一致的。

**结论：W2NER 的 `b/e` 与 ProcNet 的 `[sent_idx, start, end]` 是同一口径，可以直接转换。**

---

### 4. 关于 `end` 边界约定

**判断：这一项已经对齐，没有问题。**

你确认了：

- 采用的是 **左闭右开** `[start, end)`
- W2NER 中 `e = token_indices[-1] + 1`
- ProcNet 侧也按 `sentence[start:end]` 切片理解

**结论：边界约定一致，不构成耦合风险。**

---

### 5. 关于 `sentence` 的粒度

**判断：这一项已经对齐，没有问题。**

你说明：

- `sentence` 是原始字列表
- 不是 BERT subword 列表
- tokenizer 是单独过程，不影响这个字段

这意味着：

- 位置索引不会受到 subword 展开影响
- `"".join(sentence)` 与字符偏移是一一对应的

**结论：这一项是成立的，也是你当前方案能顺利耦合的根本前提之一。**

---

### 6. 关于 ProcNet 读取配置

**判断：你当前不是在跑 pseudo dataset 分支，而是在正常读取正式格式数据。**

你补充的配置显示：

```python
--read_pseudo=False
```

这说明：

- 你不能指望“伪数据宽松读取逻辑”来兜底
- 你的输入格式必须真正符合正式 DocEE/ProcNet 读取流程的要求

**结论：这一点很关键。它意味着你现在的转换格式必须严格自洽，尤其是实体 key、field、事件结构三者必须统一。**

## 结合你的仓库与官方仓库后的最终判断

### A. 索引层面：成立

**明确判断：索引口径没有问题。**

原因：

- W2NER 输出是字级
- `b/e` 是字符偏移
- ProcNet 也用字符切片校验 span
- 两边都用 `[start, end)`

**结论：你当前方案在“位置对齐”上是正确的。**

---

### B. 文档聚合层面：成立

**明确判断：按 `doc_id` 聚合、按 `sent_id` 排序的设计是正确的。**

原因：

- W2NER 是句子级输出
- ProcNet 是文档级输入
- 你已经把同一文档的句子重新聚合回文档结构
- `doc_id` 数量和集合也已经对上了

**结论：你当前方案在“句子 → 文档”的重建上是正确的。**

---

### C. 实体 key 设计：可用，但属于“修改版协议”

**明确判断：你当前的复合 key 设计可以解决冲突问题，但已经偏离官方最自然的 span-key 语义。**

这不是绝对错误，但意味着：

- 你的修改版 ProcNet 必须整体接受这种 key
- 不能只在数据写入阶段改 key，而读取、事件映射、训练、评估阶段仍按旧假设处理

**结论：这是“可行但要全链路一致”的设计。**

---

### D. 类型映射：有风险，需要持续核查

**明确判断：类型大小写和命名规范不是小问题，而是实际风险点。**

你已经出现了这类映射：

- W2NER：`orderapp`
- ProcNet：`orderApp`

以及：

- W2NER：`date`
- ProcNet：`startDate`

这说明你不是做“同名直传”，而是在做“类型 → 事件字段”的映射。

这会引出两个问题：

1. **这是不是稳定的一一映射？**
2. **W2NER 的实体类别是否真的等价于 ProcNet 的事件字段？**

例如：

- `departurestation` → 你样例里最终变成了 `name`
- `首都航空机场` → ProcNet 样例里却是 `首都航空`

这说明你这里不只是“字段名改写”，还存在**实体语义裁剪 / 改写**。

**结论：真正的主要风险不在索引，而在字段语义映射。**

---

### E. 事件监督：不是自动成立的

**明确判断：只把实体转过去，并不自动等于完成了 ProcNet 的事件建模。**

你当前 ProcNet 样例里：

```json
"recguid_eventname_eventdict_list": [
  [0, "flight", {...}]
]
```

这说明你已经不再是“事件留空”的方案，而是实际构造了事件。

这比“空事件列表”更进一步，但也意味着要回答一个关键问题：

- 这些事件 argument 是来自原始标注，还是由 W2NER 预测实体再拼装出来的？

如果是后者，那么：

- 事件结构正确性取决于你自己的映射规则
- 这不再只是格式转换，而是“实体 → 事件”的规则构造

**结论：你当前方案已经不仅是数据格式耦合，而是进入了事件语义重建阶段。这里必须单独审查规则。**

## 最后的明确结论

### 可以直接确认正确的部分

1. **W2NER 的 `b/e` 与 ProcNet 的 `start/end` 口径一致**
2. **两边的 span 边界规则一致，都是左闭右开**
3. **`sentence` 是字级列表，因此字符偏移可直接复用**
4. **按 `doc_id` 聚合句子恢复文档的方向是正确的**
5. **从 W2NER 实体生成 ProcNet 的 span 位置信息是可行的**

---

### 目前真正需要重点盯的部分

1. **实体类型名与 ProcNet field 名是否一一稳定对应**
2. **复合 key 设计是否在你的修改版 ProcNet 全链路中都被接受**

4. **是否存在预测实体文本与最终事件字段文本不一致的情况**
5. **是否存在“W2NER 预测实体”到“ProcNet 事件字段”之间的人为语义改写**
